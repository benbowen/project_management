# Agent Skills — Project Management Board

This file tells an agent how to interact with the Kanban board over HTTP. The agent should **not** edit `data/*.json` directly — go through the API so the running UI stays in sync.

Base URL: `http://localhost:5001`

All requests/responses are JSON. The server must be running (`./start.sh`).

## Data model

**Project**

```json
{
  "id": "snake_case_id",
  "name": "Display name",
  "description": "",
  "repo": "",
  "status": "active",
  "last_updated": "YYYY-MM-DD",
  "columns": {
    "in_progress": [],
    "ready_for_review": [],
    "next": [],
    "later": [],
    "long_term": [],
    "recent": [],
    "completed": []
  },
  "archived": [],
  "notes": "",
  "session_log": [{"date": "YYYY-MM-DD", "summary": "..."}]
}
```

**Card**

```json
{
  "id": "8charhex",
  "title": "Short title",
  "description": "Body",
  "plan_path": "path/to/PLAN.md",
  "due_date": "YYYY-MM-DD",
  "created": "YYYY-MM-DD",
  "completed_at": "YYYY-MM-DDTHH:MM:SS",
  "tags": ["tag1", "tag2"]
}
```

`id` is generated by the server on create — never set it client-side.

`repo` on a project should be the **absolute path** to the project folder on disk (e.g. `/home/user/SourceCode/my-project`). Agents can `cd` into it directly to read code, run commands, or resolve relative `plan_path` values on cards.

`due_date` is optional (`YYYY-MM-DD`). Set it when a card has a real deadline. The UI highlights overdue and due-today cards.

`plan_path` is optional and free-form. Set it when there's a planning/design doc on disk that goes with the card — typically a `PLAN.md` or similar in a repo. The path can be relative (resolved against the project's `repo`) or absolute. Reading agents should treat it as the canonical source of detail when present.

## Common workflows

### Find a project

```bash
curl -s http://localhost:5001/api/projects
# → [{"id":"foo","name":"Foo","status":"active"}, ...]
```

The list endpoint returns the index only. Fetch the full project to see cards:

```bash
curl -s http://localhost:5001/api/projects/<id>
```

### Add a card to "In Progress"

```bash
curl -s -X POST http://localhost:5001/api/projects/<id>/cards \
  -H "Content-Type: application/json" \
  -d '{"column":"in_progress","title":"Wire up auth","description":"...","plan_path":"docs/auth-plan.md","tags":["backend"]}'
```

Default column is `next` if omitted. Valid columns: `in_progress`, `ready_for_review`, `next`, `later`, `long_term`, `recent`, `completed`.

When you finish a card but it needs the user to verify or accept the work, move it to `ready_for_review` rather than `completed`. Reserve `completed` for things the user has actually signed off on.

### See what's been completed recently

```bash
curl -s "http://localhost:5001/api/activity?since=2026-04-27"
# → [{"completed_at":"2026-04-27T14:32:11","project_id":"foo","project_name":"Foo","card_id":"abc12345","title":"...","tags":[...]}]
```

Returns a flat list (newest first) of cards with a `completed_at` timestamp, drawn from each project's `completed` column **and** its archived items. Omit `since` for all-time.

Cards in `completed` carry a `completed_at` ISO timestamp (`YYYY-MM-DDTHH:MM:SS`) — the server sets it when a card transitions into `completed` and clears it if the card is moved back out. Older cards completed before this field existed have no timestamp and are skipped from date-filtered queries.

### See everything waiting on user review

```bash
curl -s http://localhost:5001/api/review-queue
# → [{"project_id":"foo","project_name":"Foo","cards":[{...}]}]
```

Use this to build a "what's blocking me?" view before the user asks. Empty list = nothing pending.

### Move a card between columns

```bash
curl -s -X PUT http://localhost:5001/api/projects/<id>/cards/<card_id> \
  -H "Content-Type: application/json" \
  -d '{"column":"completed"}'
```

Same endpoint also updates `title`, `description`, `tags`. Send only the fields you want to change.

### Log what you did this session

When a chunk of work wraps up, append a session log entry. This is what shows up in `CLAUDE.md` next time.

```bash
curl -s -X POST http://localhost:5001/api/projects/<id>/log \
  -H "Content-Type: application/json" \
  -d '{"summary":"Implemented login flow; left password reset for next session."}'
```

Date is set server-side. Only the most recent 50 entries are kept.

### Archive completed cards

```bash
curl -s -X POST http://localhost:5001/api/projects/<id>/archive \
  -H "Content-Type: application/json" \
  -d '{"card_ids":["abc12345","def67890"]}'
```

Cards move to `project.archived` with an `archived_date`. They're not deleted.

### Create a new project

```bash
curl -s -X POST http://localhost:5001/api/projects \
  -H "Content-Type: application/json" \
  -d '{"name":"My Project","description":"...","repo":"~/repos/my-project"}'
```

The `id` is derived from `name` (lowercased, spaces → underscores).

## Full endpoint reference

| Method | Path | Purpose |
|---|---|---|
| GET | `/api/projects` | List active projects (index only) |
| GET | `/api/review-queue` | Cross-project list of `ready_for_review` cards, grouped by project |
| GET | `/api/in-progress` | Cross-project list of `in_progress` cards, grouped by project |
| GET | `/api/column-overview/<column>` | Cross-project list of any column's cards, grouped by project |
| PUT | `/api/projects/<id>/columns/<column>/reorder` | Set column order; body `{card_ids: [...]}` (top = highest priority) |
| GET | `/api/activity?since=YYYY-MM-DD` | Flat list of all completed cards with timestamps; `since` is optional |
| POST | `/api/projects` | Create project |
| GET | `/api/projects/<id>` | Get full project |
| PUT | `/api/projects/<id>` | Update name/description/repo/status/notes |
| DELETE | `/api/projects/<id>` | Hard-delete project |
| POST | `/api/projects/<id>/archive-project` | Archive whole project |
| GET | `/api/archived-projects` | List archived projects |
| POST | `/api/archived-projects/<id>/restore` | Restore archived project |
| POST | `/api/projects/<id>/cards` | Add card |
| PUT | `/api/projects/<id>/cards/<card_id>` | Update or move card |
| DELETE | `/api/projects/<id>/cards/<card_id>` | Delete card |
| POST | `/api/projects/<id>/archive` | Archive cards by id |
| POST | `/api/projects/<id>/log` | Append session log entry |

## Conventions for agents

- **Check the server is up first.** `curl -s -o /dev/null -w "%{http_code}" http://localhost:5001/api/projects` should return `200`. If not, tell the user to run `./start.sh`.
- **Match existing tags.** Before adding a card, fetch the project and reuse tag strings already in use rather than inventing new ones.
- **Don't fabricate ids.** Card ids and dates come from the server.
- **Keep titles short.** Card titles are display-width-limited in the UI; aim for under 60 chars and put detail in `description`.
- **Log at session end.** A one-sentence `session_log` entry is far more valuable than perfect card hygiene — it's what gives the next session context.
